## 1. 模板遵循规则

如果项目的 `templates/` 目录中存在模板文件，必须遵循模板结构：

| 页面类型 | 对应模板 | 遵循规则 |
|----------|----------|----------|
| 封面 | `01_cover.svg` | 继承背景、装饰元素、布局结构；替换占位内容 |
| 章节页 | `02_chapter.svg` | 继承编号风格、标题位置、装饰元素 |
| 内容页 | `03_content.svg` | 继承页眉/页脚风格；**内容区域可自由布局** |
| 尾页 | `04_ending.svg` | 继承背景、致谢语位置、联系信息布局 |
| 目录页 | `02_toc.svg` | **可选**：继承目录标题、列表风格 |

### 页面-模板映射声明（必须输出）

生成每一页之前，必须明确输出使用了哪个模板（或"自由设计"）：

```
📝 **模板映射**：`templates/01_cover.svg`（或"无（自由设计）"）
🎯 **遵循规则 / 布局策略**：[具体描述]
```

- **内容页**：模板仅定义页眉和页脚；内容区域由执行器自由布局
- **无模板**：完全按照设计规范与内容大纲生成

## 2. 设计参数确认（强制步骤）

> 在生成第一张 SVG 页面之前，**必须审阅设计规范与内容大纲中的关键设计参数**，确保后续所有生成严格遵循规范。

必须输出确认内容包括：画布尺寸、正文字号、配色方案（主色/辅色/强调色 HEX 值）、字体方案。

**为什么这一步是强制的？** 防止"规范写的是一回事，执行做的是另一回事"的断层。

## 反 AI 味设计规则（每页检查）

以下设计模式是 AI 生成的"平庸美学"特征。除非用户/品牌明确要求，否则**禁止使用**：

| 避免 | 为什么是 AI 味 | PPT 中的表现 | 唯一例外 |
|------|----------------|--------------|----------|
| 紫色渐变背景（#7C3AED→#4F46E5 色系） | AI 训练语料中的"科技感"公式 | 标题栏渐变填充 | 品牌主色是紫色 |
| Emoji 当图标 | 训练语料中的列表+emoji 综合症 | 列表项使用 🚀📊✨ | 品牌风格本身使用 emoji |
| 圆角卡片 + 左侧彩色边框 | Material / Tailwind 的陈词滥调 | 带彩色左边框的内容卡片 | 用户明确要求 |
| SVG 手绘人物/场景 | AI 画的 SVG 人脸总是五官不齐 | 用 `<path>` 画人脸 | 几乎不可接受 |
| 赛博霓虹 / #0D1117 深色背景 | GitHub 深色模式的美学抄袭 | 技术演讲默认用深色主题 | 品牌方向是赛博霓虹 |
| 每个列表项配一个装饰图标 | 不传递额外信息的图标 = 视觉噪音 | 5 个列表项各配一个抽象图标 | 图标确实有助于理解 |
| 多色渐变按钮/标签 | 过度装饰 | CTA 按钮用 3 色渐变 | 品牌设计要求如此 |

**正向追求**：一个强调色贯穿全套 > 多色堆砌；一个令人惊艳的细节 > 精力平均分配到每一页。

## 每页四问（内心独白）

生成每一页的 SVG 之前，回答以下四个问题（以 XML 注释写在 SVG 文件顶部）：

1. **叙事角色**：这一页是封面英雄页 / 章节过渡 / 数据展示 / 引言留白 / 流程讲解 / 结尾行动号召？
   → 直接决定选择哪种布局模式
2. **观众距离**：1 米笔记本屏幕 / 3 米会议室投影 / 10 米报告厅？
   → 距离越远 → 字号越大、信息越少、对比度越强
3. **视觉温度**：安静克制 / 热情活力 / 冷静专业 / 权威庄重？
   → 决定色彩明度、装饰密度、留白比例
4. **容量检查**：这一页的唯一核心信息是什么？观众能否在 5 秒缩略图扫视中抓住它？
   → 装不下就拆成两页

## 品味锚点（面对等价选择时的方向性指引）

| 维度 | 偏好 | 避免 |
|------|------|------|
| 标题字体 | 有设计质感的字体（黑体 / 思源黑体） | 全场 Arial 或微软雅黑 |
| 色彩策略 | 一个暖色基调 + 一个贯穿全套的强调色 | 每页换一个强调色 / 彩虹配色 |
| 信息密度 | 更少的容器、更少的边框、更少的装饰图标 | 每条信息都包在带阴影的卡片里 |
| 细节工艺 | 选一个点做到 120%（一个惊艳的数据可视化 / 一个精致的封面） | 每页都"大差不差" |
| 留白 | 通过构图和对齐解决空旷感 | 用装饰元素填满空白区域 |
| 数据展示 | 数据 + 对比基准 + 一句话解读 | 一个孤零零的数字没有上下文 |

## 3. 执行指南

- **临近原则**：相关元素放在一起形成视觉分组；不相关分组之间增大间距以强化逻辑结构
- **绝对遵循规范**：严格遵循规范中的色彩、布局、画布格式和字体参数
- **遵循模板结构**：如果存在模板，继承模板的视觉框架
- **主 Agent 负责制**：SVG 生成必须由当前主 Agent 完成，不能委托给子 Agent，因为每一页都依赖共享的上游上下文和跨页视觉连续性
- **生成节奏**：先锁定全局设计上下文，然后在同一连续上下文中逐页顺序生成；禁止分批生成（例如每批 5 页）
- **分阶段批量生成**（推荐）：
  1. **视觉构建阶段**：按页面顺序连续生成所有 SVG 页面，确保设计风格和布局坐标的高度一致性（视觉一致性）
  2. **逻辑构建阶段**：所有 SVG 定稿后，批量生成演讲备注以确保叙事连贯性（叙事连续性）
- **技术规范**：SVG 技术约束和 PPT 兼容性规则参见 [shared-standards.md](shared-standards.md)
- **视觉深度**：使用 shared-standards.md 中的滤镜阴影、发光效果、渐变填充、虚线描边和渐变叠加创造层次纵深 — 没有立体感和强调效果的扁平页面看起来像半成品

### SVG 文件命名规范

文件命名格式：`<编号>_<页面名>.svg`

- **中文内容** → 中文命名：`01_封面.svg`、`02_目录.svg`、`03_核心优势.svg`
- **英文内容** → 英文命名：`01_cover.svg`、`02_agenda.svg`、`03_key_benefits.svg`
- **编号规则**：两位数字，从 01 开始
- **页面名**：简洁且描述性，与设计规范与内容大纲中的页面标题对应

## 4. 图标使用

> ⛔ **GATE — design_spec.md §VI 图标清单是执行器的唯一图标指令来源。** 执行器必须：
> 1. 读取 `design_spec.md` 第 VI 节的图标策略和推荐图标清单
> 2. 严格按照清单中指定的图标库和图标名使用 `data-icon` 占位符
> 3. 只有当 design_spec 明确批准 Emoji（策略 A）时，才能在 SVG 中使用 `<text>` emoji
> 4. **禁止用 SVG 基础图形（圆/方/线）"手画"来替代图标库中已有的图标** — 这是偷工减料，不是设计选择
>
> 如果 design_spec.md 缺失或第 VI 节为空，执行器必须**暂停并报告**，而非自行决定图标策略。

四种方式：**A: Emoji**（`<text>🚀</text>`）| **B: AI 生成**（SVG 基础图形）| **C: 内置图标库**（`templates/icons/` 6700+ 图标，推荐）| **D: 自定义**（用户指定）

**内置图标 — 占位符方式（推荐）**：

```xml
<!-- chunk（默认 — 直线几何、尖角、结构化） -->
<use data-icon="chunk/home" x="100" y="200" width="48" height="48" fill="#005587"/>

<!-- tabler-filled（贝塞尔曲线造型、圆润流畅轮廓） -->
<use data-icon="tabler-filled/home" x="100" y="200" width="48" height="48" fill="#005587"/>

<!-- tabler-outline（轻量线条风格 — 仅屏幕展示的演示） -->
<use data-icon="tabler-outline/home" x="100" y="200" width="48" height="48" fill="#005587"/>
```

> 无需手动运行 `embed_icons.py`；`finalize_svg.py` 后处理工具会自动嵌入图标。

**三个图标库**：

| 图标库 | 风格 | 数量 | 前缀 | 使用场景 |
|--------|------|------|------|----------|
| `chunk` | 填充 · 直线几何（尖角、方正） | 640 | `chunk/` | ✅ **默认** — 大多数场景 |
| `tabler-filled` | 填充 · 贝塞尔曲线造型（圆润流畅） | 1000+ | `tabler-filled/` | 设计要求圆润、有机的图标造型时 |
| `tabler-outline` | 描边/线条 | 5000+ | `tabler-outline/` | 仅屏幕展示的演示需要轻盈优雅美感时 |

> ⚠️ **一套演示 = 一个图标库。** 绝不混用不同图标库的图标。如果选定的图标库缺少精确匹配的图标，在**同一图标库内**找最接近的替代 — 不要跨库填补空缺。

**搜索图标** — 使用终端，零 token 消耗：
```bash
ls ${SKILL_DIR}/templates/icons/chunk/ | grep home
ls ${SKILL_DIR}/templates/icons/tabler-filled/ | grep home
ls ${SKILL_DIR}/templates/icons/tabler-outline/ | grep chart
```

**抽象概念 → 图标名**（名称以 `chunk` 为例；tabler 系列使用各自的等价名 — 用 `ls | grep` 验证）：

| 概念 | chunk | tabler-filled / tabler-outline |
|------|-------|-------------------------------|
| 增长/上升 | `arrow-trend-up` | same |
| 下降/减少 | `arrow-trend-down` | same |
| 成功/完成 | `circle-checkmark` | `circle-check` |
| 警告/风险 | `triangle-exclamation` | `alert-triangle` |
| 创新/灵感 | `lightbulb` | `bulb` |
| 战略/目标 | `target` | same |
| 效率/速度 | `bolt` | same |
| 协作/团队 | `users` | same |
| 设置/配置 | `cog` | `settings` |
| 安全/信任 | `shield` | same |
| 资金/金融 | `dollar` | `currency-dollar` |
| 时间/截止 | `clock` | same |
| 位置/区域 | `map-pin` | same |
| 沟通 | `comment` | `message` |
| 分析/数据 | `chart-bar` | same |
| 流程/循环 | `arrows-rotate-clockwise` | `refresh` |

> 一目了然的名称（home、user、file、search、arrow 等）— 直接 `grep chunk/` 即可，无需查表。

> ⚠️ **图标验证规则**：如果设计规范包含图标清单，执行器**只能**使用清单中批准的图标。使用任何图标前，通过 `ls | grep` 搜索验证其存在。**同一演示中严禁混用不同图标库的图标** — 只使用设计规范中指定的图标库。

## 5. 可视化参考

当设计规范包含 **VII. 可视化参考列表** 时，在绘制使用这些可视化类型的页面之前，从 `templates/charts/` 读取引用的 SVG 模板。路径保持为 `templates/charts/`（向后兼容）。

🚧 **GATE — 首次使用前必读**：当执行器首次遇到设计规范第 VII 节中列出的可视化类型时，**必须**在生成该页面之前 `read_file templates/charts/<chart_name>.svg`。从模板中提取布局坐标、卡片结构、间距节奏和视觉逻辑，作为**创意参考和灵感来源** — 而非严格复制。然后使用项目自身的配色方案、字体和内容独立设计页面。

> **工作流程**：读取模板 SVG → 理解结构和间距 → 参考后独立设计 SVG → 不要照搬模板。
> **复用**：某种可视化类型读取并理解后，后续相同类型的页面无需重复读取。
> **变更**：当可视化类型改变或需要重新参考结构时，读取新模板。

**适配规则**：
- **必须保留**：设计规范中指定的可视化类型（柱状图/折线图/饼图/时间轴/流程图/框架图等）
- **必须适配**：数据值、标签、颜色（匹配项目配色方案）和尺寸以适应页面布局
- **可自由调整**：视觉构图、坐标轴范围、网格线、图例位置、间距、装饰元素 — 鼓励创造性发挥，前提是图表保持准确和可读
- **禁止**：在没有设计规范依据的情况下更改可视化类型，或遗漏大纲中指定的数据点/结构元素

> 可视化模板：`templates/charts/`（52 种类型）。索引：`templates/charts/charts_index.json`

## 7. 字体使用

根据设计规范与内容大纲中的字体方案，为不同文本角色应用对应字体：

| 角色 | 中文推荐 | 英文推荐 |

**硬性最小值**：无论内容密度级别如何，正文文字绝不能小于 **18px**，标题绝不能小于 **28px**。如果内容放不下，**拆成更多页面**而不是缩小字号。

## 视觉韵律（三个维度）

单调的幻灯片是模板幻灯片，不是设计过的幻灯片。在三个维度保持韵律：

- **色彩韵律**：以浅色背景页为主 → 偶尔出现彩色章节分隔页 → 偶尔出现深色背景段落
- **密度韵律**：文字密集页 → 图片/图表主导页 → 引言/留白呼吸页
- **布局韵律**：相同布局模式连续不超过 **3 页**

## 展示验证检查点 — 2页 Demo 确认流程（v2.0 升级）

> **核心理念**：先出 2 页让用户"尝味道"，确认对味再全量生成——大幅降低方向性返工成本。

**触发条件**：总页数 **≥ 6** 时（原为 8，下调以覆盖更多场景）。

**流程**：
1. 生成**封面页**（`01_cover`）+ **最具代表性的一页内容页**（通常是信息密度最高的那页）
2. 对这两页进行自检（见下方 5 维自评审）
3. ⛔ **BLOCKING**：向用户展示 2 页预览，附带设计方向总结，然后提问：

```markdown
## 🎨 2页 Demo 预览

✅ 封面 + 典型内容页已生成。
📐 设计方向：[哲学名] / [色板描述] / [字体方案]

**对味吗？** 回复：
- "OK" → 全量生成剩余 X 页
- "字大一点 / 换个颜色 / 更克制" → 我调整后重出这 2 页
- "换个方向" → 回到方向选择
```

4. 用户确认 → 全量生成剩余页面（无需再次暂停）
5. 用户要求调整 → 修改设计参数 → 重新生成 2 页 → 再次确认

**子 Agent 例外**：当本 Skill 由子 Agent 执行时，跳过此 BLOCKING，直接全量生成。

## 8. 演讲备注生成框架

### 任务 1：生成完整演讲备注文档

在**所有 SVG 页面生成并定稿后**，进入"逻辑构建阶段"，在 `notes/total.md` 中生成完整的演讲备注文档。

**为什么不逐页生成？** 批量编写备注可以像剧本一样规划过渡衔接，确保演讲逻辑连贯。

**格式**：每页以 `# <编号>_<页面标题>` 开头，页面之间用 `---` 分隔。每页包含：脚本文本（2-5 句），`要点：① ② ③`，`时长：X 分钟`。除第一页外，每页的文本以 `[过渡]` 短语开头。

**基础舞台指示标记**（所有风格通用）：

| 标记 | 用途 |
|------|------|
| `[停顿]` | 关键内容后的留白，让观众消化吸收 |
| `[过渡]` | 每页文本开头的独立段落，衔接上一页 |

> 每种风格可扩展额外标记（`[互动]`/`[数据]`/`[观察]`/`[对标]` 等），参见 `executor-{style}.md`。

**语言一致性规则**：演讲备注中的所有结构标签和舞台指示标记**必须匹配演示内容的语言**。当演示内容为非英文时，本地化每个标签 — 不要在非英文内容中混入英文标签。

| English | 中文 | 日本語 | 한국어 |
|---------|------|--------|--------|
| `[Transition]` | `[过渡]` | `[つなぎ]` | `[전환]` |
| `[Pause]` | `[停顿]` | `[間]` | `[멈춤]` |
| `[Interactive]` | `[互动]` | `[問いかけ]` | `[상호작용]` |
| `[Data]` | `[数据]` | `[データ]` | `[데이터]` |
| `[Scan Room]` | `[观察]` | `[観察]` | `[관찰]` |
| `[Benchmark]` | `[对标]` | `[ベンチマーク]` | `[벤치마크]` |
| `Key points:` | `要点：` | `要点：` | `핵심 포인트:` |
| `Duration:` | `时长：` | `所要時間：` | `소요 시간:` |
| `Flex:` | `弹性：` | `調整：` | `조정:` |

> 以上未列出的语言，将每个标签翻译为该语言中对应的自然术语。

**要求**：

- 备注应口语化，行文自然流畅
- 突出每页的核心信息和演讲要点
- 用户可在 `notes/` 目录中手动编辑和覆盖

### 任务 2：拆分为单页备注文件

## 8.5 五维自评审 Gate（v2.0 新增 — 交付前强制）

> 在所有 SVG 生成完成、进入后处理之前，执行器必须对全套成品进行 5 维评分。任何维度低于 3/5 → 自动修复再交付。

### 评分维度

| # | 维度 | 满分标准 | 常见扣分原因 |
|---|------|---------|-------------|
| 1 | **哲学一致性** | 全套页面的视觉 DNA 与选定哲学完全吻合 | 中途跑偏回"默认安全风格" |
| 2 | **视觉层级** | 每页有一个明确焦点，5秒缩略图测试通过 | 多个元素争夺注意力、没有主次 |
| 3 | **执行质量** | 字体/间距/对齐/颜色值精确匹配 design_spec | 字号偏差、间距不一致、颜色抄错 |
| 4 | **内容特异性** | 每个文字/数字/图表都来自源文档，零 AI 填充 | "增长 300%"无出处、"Feature One"占位文字 |
| 5 | **克制度** | accent 色每页 ≤ 2 处，装饰元素精准且唯一 | 3+ 处强调色、多个竞争性装饰 |

### 执行流程

```
1. 逐页快速扫描，为每个维度打 1-5 分
2. 输出评分卡（内部，不发给用户）：
   哲学一致: X/5 | 视觉层级: X/5 | 执行质量: X/5 | 内容特异: X/5 | 克制度: X/5
3. 任何维度 < 3 → 定位问题页 → 修复 → 重新评分
4. 全部 ≥ 3 → 通过 Gate → 进入后处理
5. 两轮修复后仍有 < 3 → 标注问题交付用户决定
```

### Anti-AI-Slop 检查清单（PPT 专用 v2.0）

在 5 维评审中同步检查以下 AI 生成常见陋习。发现任一项 → 该页自动修复：

| # | 陋习 | 正确做法 |
|---|------|---------|
| 1 | 紫色/蓝紫渐变铺满标题栏 | 用纯色或极微妙渐变 |
| 2 | 每个列表项前面配一个装饰图标 | 仅在图标增加理解力时使用 |
| 3 | "10× faster"等无来源虚假数据 | 用源文档真实数据，没有就留 `—` |
| 4 | 所有卡片都有阴影+圆角+彩色左边框 | 统一风格但避免三件套组合 |
| 5 | 每页标题下面都有一行"描述性副标题" | 副标题仅在确实需要时使用 |
| 6 | 封面用大面积深色渐变背景 | 除非哲学明确要求，否则用浅色 |
| 7 | 正文使用 Inter/Roboto/Arial 作为 display 字体 | 它们是 body 字体，display 要有设计感 |
| 8 | 同一页出现 3+ 种不同颜色的强调 | 单一 accent，最多 2 处 |
| 9 | 空白区域用装饰性几何图形填充 | 留白即设计，用构图解决空旷 |
| 10 | emoji 替代图标库图标 | 除非设计方向明确允许 emoji |

## 8.6 Token Lint 校验（五维评审后、交付前强制）

> 在 §8.5 五维自评审通过后、进入后处理之前，执行器必须对全套成品执行 Token Lint 校验。任何校验项不通过 → 自动修复后再交付。

### 校验项

| # | 校验项 | 标准 | 不通过处理 |
|---|--------|------|-----------|
| 1 | **色板对比度** | fg on bg ≥ 4.5:1（WCAG AA）；accent on bg ≥ 3:1 | 调深 fg 或调浅 bg 直到达标 |
| 2 | **Token 引用完整性** | design_spec.md 中所有 `{xxx}` 引用在品牌 token 的 YAML frontmatter 中都能找到对应路径 | 缺失引用 → 使用 default.md 对应值替补并标注 |
| 3 | **字体一致性** | SVG 中 `font-family` 属性值与 token 的 `typography.display/body/mono.fontFamily` 声明一致 | 替换为 token 声明的字体 |
| 4 | **accent 用量** | 每页 SVG 中 accent 色（`{colors.accent}`）出现次数 ≤ `{ppt.accentBudget}` | 移除多余 accent，降级为 secondary 或 muted |

### 对比度计算方法

```
相对亮度 L = 0.2126*R + 0.7152*G + 0.0722*B  （R/G/B 为线性化后的 0-1 值）
对比度 = (L_lighter + 0.05) / (L_darker + 0.05)
```

### 执行流程

```
1. 从品牌 token 提取 colors、typography、ppt.accentBudget
2. 计算 primary-on-background 和 accent-on-background 对比度
3. 扫描 design_spec.md 中所有 {xxx} 引用，验证路径存在性
4. 扫描每页 SVG，统计 font-family 使用 & accent 色出现次数
5. 任何不通过 → 输出问题清单 → 自动修复 → 重新验证
6. 全部通过 → 进入后处理
```

### 输出格式（内部日志）

```
🔍 Token Lint 结果：
  [✅/❌] 对比度：fg/bg = X.X:1, accent/bg = X.X:1
  [✅/❌] 引用完整性：N/N 引用已解析
  [✅/❌] 字体一致性：display/body/mono 均匹配
  [✅/❌] accent 用量：最大 N 处/页（限制 M）
```
